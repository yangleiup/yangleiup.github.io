# 项目经历

# 数据预加载缓存 字节

  • 诉求：计算数据较慢，减少页面操作等待时间
  • 方案：
    • 请求代理，读写入sessionStorage
    • 无主动请求操作，进行预加载请求队列中的请求操作
  • 效果：
    • 用户使用满意度评分由3分涨到4.5分（5分满分）

# 多业务方配置系统

  • 诉求：多业务方快速接入
  • 系统：
    • 人群洞察字节
    • 工单系统、任务系统贝壳
  • 方案：
    • 提供 一套页面
      • 不同业务配置洞察条件、洞察图标人群洞察
      • 列表、录入、详情、处理工单系统、任务系统
    • 根据业务配置 显示页面、逻辑
    • 配置上线下线功能
  • 难点：
    • 超级表单、模版模块
    • core + plugin 框架，业务组件解耦设计
  • 效果：
    • 前端没有或很少开发量
    • 人群洞察：模块配置，系统上线后一个双月支持4个行业上线，半年通用版本和行业版本支持维护了8个行业需求
    • 任务、工单系统：
      • 前端没有或很少开发量，快速接入15+个业务方
      • 支持 备件审核、房源审核、带看审核、带看回访、客诉回访、vr审核、评论审核、内容标注、关键词审核等
      • 支持 it工单、服务台工单、反馈工单、举报、纠纷、客诉等

# 技术栈升级、项目重构 贝壳

  • 诉求：
    • 老系统庞大无法在没有人力条件下完全迁移
    • 用新技术栈开发业务
    • 多业务方尽量少的代码互相影响
  • 方案：
    • 微服务架构，开发新业务
    • 旧系统，作为主系统，通过单独代理页面，路由分发控制实现iframe接入业务页面
  • 难点：
    • 主系统、业务系统 点对点通信
    • 路由 控制。 hash不同的相同路由，渲染不同dom
  • 效果：
    • 业务无感知下，新业务技术栈得到升级
    • 业务项目解耦
    • 新项目不受旧业务技术栈束缚（hooks,ts，webpack... 我们都没有太多成本的在新业务中使用）
