# 工单中台，配置系统

  • 诉求：多业务快速接入
  • 方案：
    • 提供 列表、录入、详情、处理 一套页面
    • 根据业务配置 显示页面、逻辑
  • 难点：
    • 超级表单、模版模块
    • core + plugin 框架，业务组件解耦设计
  • 效果：
    • 接入15个业务方，前端没有或很少开发量

# 技术栈升级、项目重构

  • 诉求：
    • 老系统庞大无法在没有人力条件下完全迁移
    • 用新技术栈开发业务
    • 多业务方尽量少的代码互相影响
  • 方案：
    • 微服务架构，开发新业务
    • 旧系统，作为主系统，通过单独代理页面，路由分发控制实现iframe接入业务页面
  • 难点：
    • 主系统、业务系统 点对点通信
    • 路由 控制。 hash不同的相同路由，渲染不同dom
  • 效果：
    • 业务无感知下，新业务技术栈得到升级
    • 业务项目解耦
    • 新项目不受旧业务技术栈束缚（hooks,ts，webpack4... 我们都没有太多成本的在新业务中使用）
